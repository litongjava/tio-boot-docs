## Manim Code Generation Rules
 
### [Manim Code Generation Rules: The Background Must Be Set to Pure White]

Background must be pure white, and the default font must be black.

1. **Background Color Requirement**  
   The background of all scenes must be set to pure white; no other color is allowed.

2. **Code Example**  
   In every Manim scene, the following code must be included to explicitly specify a white background:
   ```python
   self.camera.background_color = WHITE  # Set white background
   ```
   
### [ Manim Code Generation Rules Using Manim’s `Code` mobject]

you can embed source‑code snippets with syntax highlighting directly into your animation scenes. This document explains how to configure `Code` and lists which attributes are supported (any others should be avoided).

1. Usage Example

```python
        code_string = """#include <iostream>

int main() {
    int i = 0; // 初始化
    while (i < 5) { // 条件判断
        std::cout << i << std::endl; // 循环体
        ++i; // 更新变量
    }
    return 0;
}"""

        code = Code(
            code_string=code_string,
            language="cpp",
            background="rectangle",
            formatter_style="monokai",
            add_line_numbers=True,
            line_numbers_from=1,
            background_config={"fill_color": code_bg_color, "fill_opacity": 1, "stroke_width": 1, "stroke_color": BLACK}
        ).scale(0.75)
```

2. Code Supported Attributes

Any attribute not listed here is **not** supported and should be omitted when generating code.

| Property             | Type                           | Default       | Description                                                                                                      |
|----------------------|--------------------------------|---------------|------------------------------------------------------------------------------------------------------------------|
| `code_file`          | `StrPath` / `None`             | `None`        | Specifies the path to an external source file (mutually exclusive with `code_string`).                          |
| `code_string`        | `str` / `None`                 | `None`        | Directly provides the multi-line source code as a string.                                                        |
| `language`           | `str` / `None`                 | `None`        | Name of the language recognized by Pygments (e.g., `"python"`, `"cpp"`, `"java"`).                               |
| `formatter_style`    | `str`                          | `"vim"`       | Pygments color/style theme (see the list of Pygments styles).                                                    |
| `tab_width`          | `int`                          | `4`           | Number of spaces to display for each tab character (`\t`).                                                       |
| `add_line_numbers`   | `bool`                         | `True`        | Whether to display line numbers alongside the code.                                                              |
| `line_numbers_from`  | `int`                          | `1`           | Starting value for line numbers (only used if `add_line_numbers=True`).                                           |
| `background`         | `"rectangle"` / `"window"`     | `"rectangle"` | Background shape: `"rectangle"` draws a plain box; `"window"` draws a window with a title bar; set to `False` for no background. |
| `background_config`  | `dict[str, Any]` / `None`      | `None`        | Configuration details for the background appearance (e.g., border color, corner radius); only applies if `background` is not `False`. |
| `paragraph_config`   | `dict[str, Any]` / `None`      | `None`        | Typographic settings for multi‑paragraph text (e.g., line spacing, alignment), for more complex paragraph layouts. |

### 【Manim Code Generation Rule: Use `code_string` Instead of `code`】

1. **Problem Description**  
   A `TypeError` occurs because the `Code` constructor received an unexpected keyword argument `code`.

2. **Reason**  
   The `Code` mobject does not define a `code` parameter; the correct keyword for providing inline source text is `code_string`.

3. **Correct Practice (Must Follow)**  
   Always supply your source snippet via the `code_string` argument. Do **not** use `code` or any other unsupported keyword.

4. **Correct Code Example**  
   ```python
   code_block = Code(
       code_string=r"""while (condition) {
    // code block to be executed
}""",
       language="cpp",
       background="rectangle",
   )
   ```

### 【Manim Code Generation Rule: Unsupported `font_size` Parameter in `Code` Mobject】

1. **Problem Description**  
   A `TypeError` is raised when instantiating a `Code` mobject because the constructor received an unexpected keyword argument `font_size`.

2. **Reason**  
   The `Code` mobject initializer does not define a `font_size` parameter—only the documented attributes (e.g., `code_string`, `language`, `formatter_style`, etc.) are accepted.

3. **Correct Practice (Must Follow)**  
   Only pass the supported keyword arguments listed in the Manim Code Generation Rules. To adjust the visual size of the code block, use scene-level transforms (like `.scale()`) or configure supported style options such as `formatter_style` and `paragraph_config`.

4. **Correct Code Example**  
   ```python
   code = Code(
       code_string=r"""while (condition) {
    // code block
}""",
       language="cpp",
       background="rectangle",
       formatter_style="github-dark",
       # no font_size parameter here
   ).scale(1.2)  # adjust overall size after creation
   ```

### 【Manim Code Generation Rule: Accessing Lines and Parts within a `Code` Mobject】

1.  **Problem Description**
    An `AttributeError: Code object has no attribute 'code'` occurs when attempting to access individual lines or parts of the code content within a `Code` Mobject instance using an attribute named `.code` (e.g., `my_code_object.code[index]`).

2.  **Reason**
    The Manim `Code` Mobject does not expose its internal structure or rendered lines through a public attribute named `.code`. While the `Code` object processes the input code string or file, the resulting line Mobjects (often `Text` or similar) are typically managed internally, often making the `Code` object itself behave like a `VGroup` of these lines. Attempting to access `my_code_object.code` directly leads to an `AttributeError` because this attribute does not exist.

3.  **Correct Practice (Must Follow)**
    - To access a specific line within a `Code` Mobject, treat the `Code` object itself as a sequence (like a `VGroup`) and use direct indexing: `line_mobject = my_code_object[line_index]` (where `line_index` is the 0-based index of the desired line).
    - To access specific text *within* a retrieved line, first get the line Mobject using indexing, and then use methods like `.get_part_by_text("substring")` or `.get_parts_by_tex("substring")` *on that line Mobject*.
    - **Do not** attempt to access lines or parts via `my_code_object.code`.

### 【Manim Code Generation Rule: Safely Accessing Lines within a `Code` Mobject】
1.  **Problem Description**
    An `IndexError: list index out of range` occurs when attempting to access a specific line within a `Code` Mobject instance using numerical indexing (e.g., `my_code_object[line_index]`).

2.  **Reason**
    The `Code` Mobject internally parses the input code string and typically creates a sequence of sub-mobjects, often representing individual lines. However, the exact number of sub-mobjects created might not perfectly correspond to the visual number of lines in the original string, especially considering syntax highlighting, potential empty lines, or internal parsing details. Accessing an index that is greater than or equal to the *actual* number of sub-mobjects created by Manim results in an `IndexError`. In the specific case leading to the error, the code attempted to access `incorrect_code[3]`, assuming it corresponded to the fourth visual line, but the `Code` object likely had fewer than 4 sub-mobjects in its internal list.

3.  **Correct Practice (Must Follow)**
    - **Verify Length Before Indexing:** Before accessing a specific line using `my_code_object[index]`, **must** check if the index is valid by comparing it against the actual number of sub-mobjects: `if index < len(my_code_object): ...`. Print `len(my_code_object)` during debugging if unsure about the structure.
    - **Use Text-Based Selection (More Robust):** When possible, prefer selecting code parts based on their content rather than their line number. Use methods like `my_code_object.get_part_by_text("substring")` or `my_code_object.get_parts_by_text("substring")` to find the Mobject corresponding to specific code text. This is less prone to breaking if the code formatting or line count changes slightly.
    - **Iterate Safely:** If you need to process lines sequentially, iterate directly: `for line_mob in my_code_object: ...`.

4.  **Correct Code Example**
    (Illustrating safe indexing and text-based selection)

### 【Manim Code Generation Rule: Use Correct Constants for `arrange()` Alignment Direction】
1.  Problem Description
    A `NameError: name 'CENTER' is not defined` occurs when attempting to use `CENTER` as the value for the `aligned_edge` parameter in the `Mobject.arrange()` or `VGroup.arrange()` method.

2.  Reason
    The `arrange()` method's `aligned_edge` parameter expects a direction vector (like `UP`, `DOWN`, `LEFT`, `RIGHT`, or `ORIGIN`) to specify which edge of the bounding boxes should be aligned perpendicular to the arrangement direction. `CENTER` is typically defined as the coordinate point `[0, 0, 0]` in Manim, but it is not a predefined constant intended for use as an alignment _direction_ in the `aligned_edge` parameter, nor is it automatically available in the global scope for this purpose. Using an undefined name results in a `NameError`. Furthermore, the default alignment behavior when arranging vertically (like `DOWN`) is already center alignment along the perpendicular axis (horizontal center alignment), making `aligned_edge=CENTER` unnecessary even if it were defined.

3.  Correct Practice (Must Follow)
    *   When arranging Mobjects vertically (e.g., `my_group.arrange(DOWN)`) and desiring the default horizontal center alignment, **do not** specify the `aligned_edge` parameter at all. Rely on the default behavior.
    *   Similarly, when arranging horizontally (`RIGHT` or `LEFT`) and desiring vertical center alignment, omit `aligned_edge`.
    *   If you need to align along a specific edge perpendicular to the arrangement direction, use the correct direction vector constants: `UP`, `DOWN`, `LEFT`, `RIGHT`. For example, to align the left edges when arranging vertically (`DOWN`), use `aligned_edge=LEFT`.
    *   **Do not** use `CENTER` as the value for the `aligned_edge` parameter.

4.  Correct Code Example

    ```python
    from manim import VGroup, Text, DOWN, LEFT, RIGHT, ORIGIN # Necessary imports

    # Assume conclusion_text_1, conclusion_text_2, conclusion_text_3 are defined Text objects
    conclusion_text_1 = Text("Line 1")
    conclusion_text_2 = Text("Line 2, slightly longer")
    conclusion_text_3 = Text("Line 3")

    # --- Incorrect Code (Causes NameError) ---
    # conclusion_group_bad = VGroup(conclusion_text_1, conclusion_text_2, conclusion_text_3).arrange(DOWN, buff=0.5, aligned_edge=CENTER)

    # --- Correct Code (Default Horizontal Center Alignment) ---
    # Arrange vertically, horizontal alignment defaults to center
    conclusion_group_good = VGroup(conclusion_text_1, conclusion_text_2, conclusion_text_3).arrange(DOWN, buff=0.5)

    # --- Correct Code (Explicit Left Alignment - if needed) ---
    # Arrange vertically, explicitly align left edges
    # conclusion_group_left = VGroup(conclusion_text_1, conclusion_text_2, conclusion_text_3).arrange(DOWN, buff=0.5, aligned_edge=LEFT)

    # --- Correct Code (Explicit Right Alignment - if needed) ---
    # conclusion_group_right = VGroup(conclusion_text_1, conclusion_text_2, conclusion_text_3).arrange(DOWN, buff=0.5, aligned_edge=RIGHT)
    ```

### Horizontal Arrangement

When arranging horizontally (e.g., using `arrange(RIGHT)`) to achieve vertical center alignment, you can rely on the default behavior without explicitly specifying `aligned_edge=CENTER`. This is because the `aligned_edge` parameter of the `Mobject.arrange()` method only accepts direction vectors (like `UP`, `DOWN`, `LEFT`, `RIGHT`) or `ORIGIN` (for center alignment), while `CENTER` is a coordinate point `[0, 0, 0]`, which does not meet the requirement and will cause an error.

Here is the correct example code:

```python
from manim import VGroup, Square, Circle, RIGHT # Necessary imports

# Correct Example:
square = Square()
circle = Circle().next_to(square, RIGHT)
mobject_group = VGroup(square, circle)

# This will arrange them horizontally and align their vertical centers by default
mobject_group.arrange(RIGHT, buff=0.5)
```

### Manual List Creation

You **must** create lists manually. This is because Manim may encounter encoding issues when trying to write text containing Chinese characters into a LaTeX file (used for parts of `BulletedList`, likely the default bullet points). Although `Text` objects handle Chinese rendering correctly using Pango, the default implementation of `BulletedList` relies on LaTeX to create the bullet points (•, often `\cdot`), leading to conflicts.
Therefore, the generated code needs to use `VGroup` to replace `BulletedList`, where each line contains a `Text` bullet point and a `Text` list item. This avoids LaTeX encoding problems.

```Correct code example:
        from manim import * # Import necessary classes

        # Define colors (example)
        MY_WHITE = "#FFFFFF"
        config.frame_width = 16 # Example value
        config.frame_height = 9 # Example value

        class ManualListScene(Scene):
            def construct(self):
                title = Text("Course List", font_size=48).to_edge(UP)
                self.add(title) # Add title to scene

                # --- Manually create the list ---
                course_items_text = [
                    "Channelization Engineering", # Example English text
                    "Port Planning and Layout",
                    "Waterway Regulation",
                    "Engineering Project Management",
                    "Port and Coastal Hydraulic Structures",
                ]
                course_list_group = VGroup()
                item_font_size = 36
                bullet_color = MY_WHITE
                text_color = MY_WHITE
                line_buff = 0.4 # Spacing between lines
                bullet_text_buff = 0.2 # Spacing between bullet and text

                for item_text in course_items_text:
                    # Create bullet (using Text) and text item
                    bullet = Text("• ", font_size=item_font_size, color=bullet_color)
                    text = Text(item_text, font_size=item_font_size, color=text_color)
                    # Arrange bullet and text horizontally
                    line = VGroup(bullet, text).arrange(RIGHT, buff=bullet_text_buff)
                    course_list_group.add(line)

                # Arrange all lines vertically, aligned to the left
                course_list_group.arrange(DOWN, buff=line_buff, aligned_edge=LEFT)
                # Position the list (consistent with previous BulletedList positioning logic)
                course_list_group.next_to(title, DOWN, buff=0.8).align_on_border(LEFT, buff=1.0) # Align to left border with buffer
                # Fine-tune horizontal position if needed (example: shift slightly right from border)
                # course_list_group.shift(RIGHT * 1.0)

                self.play(FadeIn(course_list_group))
                self.wait()
```
### 【Manim Code Generation Rule: LaTeX Unicode/CJK Character Error due to Missing Font】
1.  **Problem Description:**
    The Manim code generation fails during the rendering of `MathTex` or `Tex` objects containing non-ASCII characters, particularly CJK (Chinese, Japanese, Korean) characters like "截距". The error log typically shows a LaTeX compilation failure (indicated by messages like "xelatex error converting to xdv" or similar `ValueError` from Manim's TeX processing) and includes LaTeX messages about missing font files (e.g., `! I can't find file '...'`, `mktextfm failed`).

2.  **Reason:**
    Manim relies on an external LaTeX compiler (`xelatex` or `lualatex` when using a custom template for CJK) to render `MathTex` and `Tex` objects. While the custom `TexTemplate` correctly specifies a Unicode-aware compiler and the `xeCJK` package, the `\setCJKmainfont{YourFontName}` command in the LaTeX preamble requires that the specified font (`Songti SC` in this case) is *actually installed and accessible* to the LaTeX distribution on the system where Manim is being run. The error indicates that the LaTeX compiler could not find or process the font files necessary for the specified CJK font, leading to the compilation failure.

3.  **Correct Practice (Must Follow):**

    *   When generating `MathTex` or `Tex` code that includes non-ASCII characters (especially CJK characters):
        *   **Define a custom `TexTemplate`**.
        *   Set the `tex_compiler` attribute of the template to `"xelatex"` (recommended) or `"lualatex"`.
        *   Set the `output_format` accordingly (typically `".xdv"` for `xelatex`).
        *   Modify the `preamble` attribute of the template to include necessary packages for Unicode/CJK support. For CJK with `xelatex`, this typically involves:
            *   `\usepackage{fontspec}`
            *   `\usepackage{xeCJK}`
            *   `\setCJKmainfont{YourFontName}`: Specify a font installed on the target system that supports the required CJK characters (e.g., `Songti SC`, `Microsoft YaHei`, `Songti SC`, `Source Han Sans SC`). **The chosen font MUST be installed on the system where Manim is run.** The generated code should use a common font name but include a comment reminding the user to verify/change it if necessary.
        *   **Pass this custom `TexTemplate`** to the `MathTex` or `Tex` object during initialization using the `tex_template=` keyword argument.
    *   For `Text` objects (not `MathTex` or `Tex`), specify the font directly using the `font=` argument (e.g., `font="Songti SC"`). This does _not_ use LaTeX but relies on system fonts via Pango/Cairo. Ensure the font is installed.

4.  **Correct Code Example:**

        output_format=".xdv",
        preamble=r"""
    \usepackage{amsmath}
    \usepackage{amssymb}
    \usepackage{fontspec}
    \usepackage{xeCJK}
    \setCJKmainfont{Songti SC} % Or Microsoft YaHei, Songti SC, etc.
    """
    )

                font_size=48,
                color=BLACK # Contrasting color for white background
            )

            # Text object uses system fonts directly via Pango/Cairo
            # Ensure the font specified here is also installed
            title_chinese = Text(
                "什么是焓变?",
                font="Songti SC", # Specify font directly for Text
                font_size=60,
                color=BLACK # Contrasting color
            )

            self.add(VGroup(title_chinese, formula_chinese).arrange(DOWN, buff=1))

### 【Manim Code Generation Rule: `Axes.get_vertical_line` Expects a Point Argument】

    *   To change the width of the camera's view (effectively zooming), use: `self.camera.frame.set_width(new_width)`.

### 【Manim Code Generation Rule: Handling Backslashes and Line Continuations in LaTeX Strings】
1.  **Problem Description:**
    When defining strings containing LaTeX code in Python for Manim, two common issues can arise:
    *   `SyntaxWarning: invalid escape sequence '\...'` (e.g., `\c` in `\circ`, `\f` in `\frac`, `\b` in `\beta`). This warning appears when a standard Python string (not explicitly a raw string) contains a backslash followed by a character, and this combination (`\c`, `\f`, etc.) is not a recognized Python escape sequence.
    *   `SyntaxError: unexpected character after line continuation character`. This error occurs if a backslash (`\`) is used at the end of a physical line in the source code to indicate that the string literal continues on the next line, but one or more characters (typically whitespace like a space or tab) are present immediately after this backslash on the same line.

2.  **Reason:**
    *   **Invalid Escape Sequence (`SyntaxWarning`):** Python's string parser, by default, interprets backslashes as initiating escape sequences (e.g., `\n` for newline, `\t` for tab). LaTeX commands, which are fundamental to `MathTex` and `Tex` mobjects, also heavily use backslashes (e.g., `\frac`, `\text`, `\alpha`). If a LaTeX command like `\circ` is written in a standard Python string, Python attempts to interpret `\c` as an escape sequence. Since `\c` is not a valid Python escape sequence, a `SyntaxWarning` is issued. While the code might sometimes run if Python passes the `\` and `c` literally, it's unreliable and bad practice.
    *   **Line Continuation Error (`SyntaxError`):** In Python, a backslash (`\`) at the very end of a physical line signifies that the current logical line of code (e.g., a long string definition) continues onto the next physical line. Python expects a newline character to immediately follow this line-continuing backslash. If any other character, including spaces or tabs, exists on the line after the backslash, it breaks this rule, leading to a `SyntaxError`.

3.  **Correct Practice (Must Follow):**
    *   **Always Use Raw Strings for LaTeX:** To prevent Python from misinterpreting LaTeX backslash sequences, **must** enclose all strings intended for LaTeX processing (i.e., passed to `MathTex`, `Tex`, etc.) in raw string literals.
        *   If the string does not require f-string formatting (variable insertion or `{{` `}}`), use `r"..."`. Example: `MathTex(r"\frac{a}{b}")`.
        *   If f-string formatting is needed, use raw f-strings: `rf"..."`. Example: `MathTex(rf"\text{{Value: }} {my_var}")`.
        This practice ensures that backslashes are treated as literal characters by Python and are passed correctly to the LaTeX compiler, resolving the `SyntaxWarning`.

    *   **Manage Long Strings Correctly:**
        *   **Line Continuation (`\`):** If you choose to split a long string literal across multiple physical lines using a backslash, you **must** ensure that the backslash is the **absolute last character** on that line. No spaces, tabs, or comments should follow it on the same line before the newline.
            ```python

            long_latex_ok = rf"\text{{This is a very long LaTeX string that needs }} \
" # Backslash is last
                          rf"\text{{to be continued on the next line.}}"
            ```
        *   **Preferred: Implicit String Concatenation:** For better readability and to avoid line continuation issues, use Python's implicit string literal concatenation. Simply place multiple string literals (raw, f-string, or raw f-string as appropriate) next to each other, typically enclosed in parentheses for grouping. Python will automatically join them.
            ```python

            long_latex_preferred = (
                rf"\text{{This is a very long LaTeX string that needs }}"
                rf" \text{{to be continued on the next line with a space between.}}"
            )
            ```
        *   **Triple-Quoted Raw Strings:** For very long blocks of LaTeX, you can use triple-quoted raw strings (e.g., `r"""..."""` or `rf"""..."""`). Newlines within these strings are preserved as newline characters in the string content. LaTeX generally handles these extra newlines well.

### 【Manim Code Generation Rule: Escaping Curly Braces in f-strings for LaTeX】
1.  Problem Description
    A `NameError: name '...' is not defined` occurs when using an f-string to generate a LaTeX string for `MathTex` or `Tex`. The error happens when a part of the LaTeX string like `_{xy}` or `\text{abc}` is intended, but the f-string interprets `{xy}` or `{abc}` as Python variables to be interpolated, rather than literal LaTeX grouping braces. For example, `rf"a_{cm}"` in the expression `MathTex(rf"a_{cm} \approx ...")` leads to `NameError: name 'cm' is not defined` because Python attempts to find and substitute a variable named `cm`.

2.  Reason
    In Python f-strings (formatted string literals), curly braces `{}` are special characters used to enclose expressions that Python evaluates and whose results are formatted into the string. If you need to include literal curly brace characters `{` or `}` in the output string (which is frequently required for LaTeX syntax, such as for multi-character subscripts `X_{subscript}`, superscripts `X^{superscript}`, or commands like `\text{some text}` or `\frac{numerator}{denominator}`), these f-string-special curly braces must be escaped. Using single curly braces like `_{cm}` or `\text{word}` inside an f-string will cause Python to parse `cm` or `word` as expressions (typically variable names) to be evaluated. If these variables are not defined in the current Python scope, a `NameError` is raised.

3.  Correct Practice (Must Follow)
    -   When constructing LaTeX strings using Python f-strings for `MathTex`, `Tex`, or similar mobjects that process LaTeX:
        -   To include a literal left curly brace `{` in your final LaTeX string, you **must** write `{{` in the f-string.
        -   To include a literal right curly brace `}` in your final LaTeX string, you **must** write `}}` in the f-string.
    -   This applies to all instances where LaTeX requires curly braces for grouping or arguments, for example:
        -   For a multi-character subscript like `a_{cm}`, the f-string should be `rf"a_{{cm}}"`.
        -   For a LaTeX command with an argument like `\text{speed}`, the f-string should be `rf"\text{{speed}}"`.

    -   If you are interpolating Python variables *inside* LaTeX braces, ensure the LaTeX braces themselves are doubled. For example, to produce `\text{value of x}` where `x_val` is a Python variable: `rf"\text{{{x_val}}}"`.
