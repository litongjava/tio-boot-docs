# linux 编译

## 安装依赖库

```
apt install build-essential -y
apt-get install nasm -y
apt-get install pkg-config -y
```

Linux 下（使用动态库版本）：

```bash
export VCPKG_LIBRARY_LINKAGE=dynamic
vcpkg install ffmpeg[mp3lame]:x64-linux-dynamic --recurse
```

由于 linux vcpkg 不支持安装 ffmpeg 动态库,所以需要用到系统包管理安装

```
sudo apt-get install libavcodec-dev libavformat-dev libavutil-dev libswresample-dev libmp3lame-dev -y
```

## 使用 cmake build 项目

在 Linux 上使用 CMake 时分两步进行：首先进行配置生成，再进行构建。

1. **配置生成阶段**

在项目根目录下执行：

```bash
cmake -S . -B cmake-build-debug
```

切换为 release 模式

```
cmake -S . -B cmake-build-release -DCMAKE_BUILD_TYPE=Release
```

output

```
-- Found JNI: /usr/java/jdk1.8.0_411/include  found components: AWT JVM
-- AVCODEC_INCLUDE_DIR: /opt/vcpkg/installed/x64-linux/include
```

这一步将使用 vcpkg 的 toolchain 文件生成构建系统到 `cmake-build-debug` 目录。

2. **构建阶段**  
   然后执行：
   ```bash
   cmake --build cmake-build-debug --target all
   ```
   ```
   cmake --build cmake-build-release --target all
   ```
   这一步将从生成的构建目录中编译项目。

## 错误记录

### can not be used when making a shared object; recompile with -fPIC

```
[ 50%] Linking C shared library libnative_media.so
/usr/bin/ld: /opt/vcpkg/installed/x64-linux/debug/lib/libavcodec.a(hevc_sao_10bit.o): warning: relocation against `ff_pw_1' in read-only section `.text'
/usr/bin/ld: /opt/vcpkg/installed/x64-linux/debug/lib/libavcodec.a(hevcdec.o): relocation R_X86_64_PC32 against symbol `ff_h264_cabac_tables' can not be used when making a shared object; recompile with -fPIC
/usr/bin/ld: final link failed: bad value
collect2: error: ld returned 1 exit status
gmake[2]: *** [CMakeFiles/native_media.dir/build.make:119: libnative_media.so] Error 1
gmake[1]: *** [CMakeFiles/Makefile2:85: CMakeFiles/native_media.dir/all] Error 2
gmake: *** [Makefile:91: all] Error 2

```

这个错误表明你正在将没有使用 PIC 编译（-fPIC）的静态 FFmpeg 库链接进一个共享库（.so），而在 Linux 上，所有被嵌入共享库的代码都必须使用 -fPIC 编译。

解决方法有三个方向：

1. **使用共享（动态）版的 FFmpeg 库**  
   通过 vcpkg 安装 FFmpeg 时，确保安装的是共享库版本。你可以在 vcpkg 中使用动态链接的 triplet 或设置环境变量：

   ```bash
   vcpkg install ffmpeg --triplet x64-linux-dynamic
   ```

   或者设置环境变量：

   ```bash
   export VCPKG_LIBRARY_LINKAGE=dynamic
   ```

   然后重新安装 FFmpeg，这样你在链接时会使用 .so 动态库而不是 .a 静态库，从而避免这个问题。

2. **重新编译 FFmpeg 静态库时启用 -fPIC**  
   如果你必须使用静态库，则需要重新编译 FFmpeg（通过 vcpkg 或自行编译）时，确保所有目标都加上 -fPIC。这通常可以在 CMake 配置阶段设置：

   ```bash
   cmake -DCMAKE_POSITION_INDEPENDENT_CODE=ON ...
   ```

   但注意，vcpkg 默认可能不启用这一选项，因此最好使用共享库方案。

3. **使用系统库**
   因为上面两个方法失败,所以笔者这里使用系统的包管理工具 apt 安装

```
sudo apt-get install libavcodec-dev libavformat-dev libavutil-dev libswresample-dev libmp3lame-dev -y
```

总结：  
建议在 Linux 下使用共享版本的 FFmpeg 库，这样既避免了 PIC 问题，也能更好地与共享库（libnative_media.so）进行链接。

### 构建后到其他计算机上运行,其他 linux 也要安装依赖库吗?

如果你的程序是动态链接到系统中的共享库（例如通过 apt 安装的 libavcodec、libavformat 等），那么在其他 Linux 系统上运行时，如果这些系统没有相应的库，就会导致程序运行失败或者出现库缺失错误。解决方法有以下几种：

1. **要求目标系统安装相应的共享库**  
   在部署说明中注明需要安装相应版本的 FFmpeg 开发包（或相关共享库），这样系统上必须预先安装这些库。

2. **静态链接**  
   你可以选择在编译时将依赖的库静态链接进你的可执行文件，这样目标系统就不需要额外安装这些共享库。但静态链接会使生成的二进制文件变大，而且某些库可能不支持或不推荐静态链接。

3. **打包共享库**  
   另一种方法是在部署时将所有必要的共享库打包到一起，通过修改程序的运行时库路径（例如使用 RPATH 或 LD_LIBRARY_PATH）让程序加载打包的库，这样即使目标系统没有安装这些库也能正常运行。

综上，如果你选择使用系统共享库，则其他 Linux 计算机上需要安装相应的库，除非你采取静态链接或打包共享库的方式。

### 我的项目构建产物不是一个静态文件,而是一个库文件(_.ddl),(_.so).我在 windows 上复制 ddl 到 path 下即可,在 linux 上可以吗?

在 Linux 上共享库的查找机制与 Windows 不同，不能简单地将 .so 文件复制到 PATH 路径下。Linux 使用的是动态链接器（dynamic linker），它默认会在系统目录（如 /lib、/usr/lib、/usr/local/lib 等）中查找共享库，同时也会参考环境变量 LD_LIBRARY_PATH 中指定的路径。

你可以考虑以下几种方法来确保程序能找到你的 .so 文件：

1. **安装到系统目录**  
   将 .so 文件复制到系统默认的共享库目录，如 /usr/local/lib，然后运行 `ldconfig` 更新缓存，这样系统就能识别新的库。

2. **设置 LD_LIBRARY_PATH**  
   如果不想安装到系统目录，可以在运行程序前设置环境变量 LD_LIBRARY_PATH，指定包含 .so 文件的目录，例如：

   ```bash
   export LD_LIBRARY_PATH=/path/to/your/library:$LD_LIBRARY_PATH
   ./your_executable
   ```

3. **使用 rpath 或 RUNPATH**  
   在链接时，可以在可执行文件中嵌入库的路径（使用 -Wl,-rpath 选项），这样程序运行时会优先从指定路径加载共享库，无需额外设置环境变量。

总之，虽然在 Windows 上只需将 DLL 文件放在 PATH 中，但在 Linux 上你需要确保系统的动态链接器能够找到你的 .so 文件。

### 在 linux 上,这些 load 命令还需要吗?如果需要,需要替换为什么?

```java
    System.load(new File("lib", "avutil-59.dll").getAbsolutePath());
    System.load(new File("lib", "swresample-5.dll").getAbsolutePath());
    System.load(new File("lib", "avcodec-61.dll").getAbsolutePath());
    System.load(new File("lib", "avformat-61.dll").getAbsolutePath());
```

在 Linux 上，动态库的文件后缀与命名规则与 Windows 不同。Windows 使用 .dll，而 Linux 使用 .so，并且通常库名前面会有 "lib" 前缀。因此，如果你的应用需要手动加载这些本地库，并且这些库不是系统默认路径下可见的，你需要做如下修改：

例如，将

```java
System.load(new File("lib", "avutil-59.dll").getAbsolutePath());
```

替换为

```java
System.load(new File("lib", "libavutil.so.59").getAbsolutePath());
```

同理，其他几行也应替换为 Linux 下对应的动态库名称：

```java
System.load(new File("lib", "libswresample.so.5").getAbsolutePath());
System.load(new File("lib", "libavcodec.so.61").getAbsolutePath());
System.load(new File("lib", "libavformat.so.61").getAbsolutePath());
```

另外，有几点需要注意：

- **系统库路径：** 如果这些动态库已经安装在系统默认的库目录中（如 /usr/lib、/usr/local/lib 等），你可以使用 `System.loadLibrary("avutil")` 等方式加载库，而无需提供完整路径，也不需要手动加载所有依赖库。

- **打包和部署：** 如果你选择将这些库随应用程序一起分发，那么确保在 Linux 上使用上述命名方式，并且运行时设置好 LD_LIBRARY_PATH 或通过 rpath 告诉系统动态链接器如何找到这些库。

总结来说，在 Linux 上需要替换为对应的 .so 文件名，并且注意库文件的查找和加载方式与 Windows 不同。

在 Linux 系统中，如果共享库（如 libavutil、libswresample、libavcodec、libavformat 等）已经安装在系统默认的库目录（例如 /usr/lib、/usr/local/lib 等），并且你的 JNI 主库在编译时已正确链接了这些依赖库，那么系统的动态链接器会在加载你的 JNI 主库时自动加载这些依赖项。

因此，你只需要调用

```java
System.loadLibrary("native_media");
```

### 如果这些动态库已经安装在系统默认的库目录 java 中还需要 System.loadLibrary 吗?

来加载你的主 JNI 库，而不必在 Java 代码中显式调用 System.loadLibrary 加载每个依赖库。这和 Windows 下需要手动加载 DLL 是不同的，因为 Linux 的动态链接器会自动处理依赖项的加载，只要它们位于系统默认搜索路径中。
